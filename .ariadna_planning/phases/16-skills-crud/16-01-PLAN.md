**2. Create `app/controllers/skills_controller.rb`:**

Follow the established CRUD pattern from RolesController/GoalsController:

```ruby
class SkillsController < ApplicationController
  before_action :require_company!
  before_action :set_skill, only: [ :show, :edit, :update, :destroy ]

  def index
    @skills = Current.company.skills.order(:name)
    @skills = @skills.by_category(params[:category]) if params[:category].present?
    @current_category = params[:category]
    @categories = Current.company.skills.where.not(category: [nil, ""]).distinct.pluck(:category).sort
  end

  def show
    @agents = @skill.agents.order(:name)
  end

  def new
    @skill = Current.company.skills.new(builtin: false)
  end

  def create
    @skill = Current.company.skills.new(skill_params)
    @skill.builtin = false

    if @skill.save
      redirect_to @skill, notice: "#{@skill.name} skill has been created."
    else
      render :new, status: :unprocessable_entity
    end
  end

  def edit
  end

  def update
    if @skill.update(skill_params)
      redirect_to @skill, notice: "#{@skill.name} skill has been updated."
    else
      render :edit, status: :unprocessable_entity
    end
  end

  def destroy
    if @skill.builtin?
      redirect_to @skill, alert: "Built-in skills cannot be deleted."
      return
    end

    name = @skill.name
    @skill.destroy
    redirect_to skills_path, notice: "#{name} skill has been deleted."
  end

  private

  def set_skill
    @skill = Current.company.skills.find(params[:id])
  end

  def skill_params
    params.require(:skill).permit(:key, :name, :description, :markdown, :category)
  end
end
```

Key design decisions:
- `index`: category filtering via `params[:category]` using existing `by_category` scope on Skill model. `@categories` computed from actual company data (not hardcoded) so it adapts to the company's skill catalog.
- `create`: forces `builtin: false` on the model AFTER applying strong params. `builtin` is NOT in `skill_params` -- users cannot set it. This ensures CRUD-04.
- `update`: allows editing all skills (including builtin) per CRUD-03. `builtin` is not in permitted params, so it cannot be changed.
- `destroy`: checks `builtin?` and redirects with alert if true per CRUD-03. Custom skills can be destroyed per CRUD-04.
- `set_skill`: scoped to `Current.company.skills.find` for tenant isolation.
- `show`: loads `@agents` for displaying which agents have the skill assigned.

```ruby
module SkillsHelper
  SKILL_CATEGORIES = %w[leadership technical creative operations research].freeze

  def skill_category_options

  end
end
```

```erb
          skill_category_options,
          { include_blank: "None" },
          { class: "form__select" } %>
  </div>

  <div class="form__field">
    <%= f.label :description, "Short description" %>
    <%= f.text_area :description, rows: 2, placeholder: "Brief summary of what this skill enables" %>
  </div>

  <div class="form__field">
    <%= f.label :markdown, "Instruction content (Markdown)" %>
    <%= f.text_area :markdown, rows: 12, required: true,
          placeholder: "# Skill Name\n\n## Purpose\nDescribe the skill's purpose...\n\n## Instructions\n1. Step one\n2. Step two" %>
  </div>

  <div class="form__actions">
    <%= f.submit class: "btn btn--primary" %>
    <%= link_to "Cancel", skill.persisted? ? skill_path(skill) : skills_path, class: "btn btn--ghost" %>
  </div>
<% end %>
```

Key form design:
- `key` field is disabled for persisted builtin skills (key is their identity, changing it would break references). Disabled fields are NOT submitted, so the key stays unchanged. Custom skills and new skills can set the key.
- `builtin` is never shown in the form -- it is controlled server-side only.
- `category` uses a select with options from `skill_category_options` helper.
- `markdown` is a large text area (12 rows) since instructions can be lengthy.
