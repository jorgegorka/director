    Edit `app/models/role.rb`:
    ```ruby
    class Role < ApplicationRecord
      include Tenantable

      belongs_to :parent, class_name: "Role", optional: true
      has_many :children, class_name: "Role", foreign_key: :parent_id, dependent: :nullify

      validates :title, presence: true
      validates :title, uniqueness: { scope: :company_id, message: "already exists in this company" }
      validate :parent_belongs_to_same_company
      validate :parent_is_not_self
      validate :parent_is_not_descendant

      scope :roots, -> { where(parent_id: nil) }

      def ancestors
        ancestors = []
        current = parent
        while current
          ancestors << current
          current = current.parent
        end
        ancestors
      end

      def descendants
        children.flat_map { |child| [child] + child.descendants }
      end

      def root?
        parent_id.nil?
      end

      def depth
        ancestors.size
      end

      private

      def parent_belongs_to_same_company
        if parent.present? && parent.company_id != company_id
          errors.add(:parent, "must belong to the same company")
        end
      end

      def parent_is_not_self
        if parent_id.present? && parent_id == id
          errors.add(:parent, "cannot be the role itself")
        end
      end

      def parent_is_not_descendant
        if parent_id.present? && id.present? && descendants.map(&:id).include?(parent_id)
          errors.add(:parent, "cannot be a descendant of this role")
        end
      end
    end
    ```

    Key design decisions:
    - `include Tenantable` -- automatically gets `belongs_to :company` and `scope :for_current_company`
    - `has_many :children, dependent: :nullify` -- when a role is deleted, its children become root roles (re-parented to nil). This is simpler than re-parenting to the deleted role's parent, which is handled in the controller's destroy action instead.
    - Three hierarchy validations: parent must be in the same company, parent cannot be self, parent cannot be a descendant (prevents cycles)
    - `ancestors` and `descendants` methods use simple Ruby traversal. For the scale of org charts (dozens, not thousands of roles), this is sufficient. Recursive CTE can be added later if performance requires it.
    - `roots` scope returns roles with no parent (top of hierarchy)

    **Step 2: Create RolesController**

    Create `app/controllers/roles_controller.rb`:
    ```ruby
    class RolesController < ApplicationController
      before_action :require_company!
      before_action :set_role, only: [:show, :edit, :update, :destroy]

      def index
        @roles = Current.company.roles.includes(:parent, :children).order(:title)
      end

      def show
      end

      def new
        @role = Current.company.roles.new
      end

      def create
        @role = Current.company.roles.new(role_params)

        if @role.save
          redirect_to @role, notice: "#{@role.title} role has been created."
        else
          render :new, status: :unprocessable_entity
        end
      end

      def edit
      end

      def update
        if @role.update(role_params)
          redirect_to @role, notice: "#{@role.title} role has been updated."
        else
          render :edit, status: :unprocessable_entity
        end
      end

      def destroy
        # Re-parent children to the deleted role's parent before destroying
        @role.children.update_all(parent_id: @role.parent_id)
        @role.destroy
        redirect_to roles_path, notice: "#{@role.title} role has been deleted."
      end

      private

      def set_role
        @role = Current.company.roles.find(params[:id])
      end

      def role_params
        params.require(:role).permit(:title, :description, :job_spec, :parent_id)
      end
    end
    ```

    Key design decisions:
    - All queries scoped through `Current.company.roles` -- a user cannot access roles from another company even by guessing IDs
    - `destroy` re-parents children to the deleted role's parent (not nullify). This preserves hierarchy continuity. For example, deleting a CTO between CEO and Developer makes Developer report to CEO. The `update_all` runs before `destroy` to avoid the `dependent: :nullify` triggering first. Actually, since `dependent: :nullify` would set parent_id to nil, we need to re-parent BEFORE destroy. The `update_all` call bypasses callbacks which is fine for this simple parent_id update.
    - `parent_id` is permitted in strong params -- the user selects the parent from a dropdown. A blank/nil value means root role.
    - `agent_id` is NOT in permitted params -- agents are not assignable yet (Phase 4)
    - `includes(:parent, :children)` on index to avoid N+1 queries

    Create a helper method. Add to `app/helpers/roles_helper.rb` (create this file):
    ```ruby
    module RolesHelper
      def options_for_parent_select(role)
        # Get all roles in the current company except the role itself and its descendants
        excluded_ids = role.persisted? ? [role.id] + role.descendants.map(&:id) : []
        available_roles = Current.company.roles.where.not(id: excluded_ids).order(:title)
        available_roles.map { |r| [r.title, r.id] }
      end
    end
    ```

        assert_equal "Designer", role.title
        assert_equal @cto, role.parent
        assert_equal @company, role.company
        assert_redirected_to role_url(role)
      end

      test "should create root role with no parent" do

          post roles_url, params: {
            role: { title: "Advisor", description: "External advisor" }
          }
        end
        role = Role.order(:created_at).last
        assert_nil role.parent_id
      end
