4. Run `bundle install` to regenerate Gemfile.lock (this will remove pg from the lock file)

**database.yml rewrite:**
Replace the entire file with a SQLite-native configuration. The structure must be:

```yaml
# SQLite. Versions 3.38.0 and up are supported.
#   gem install sqlite3
#
# Ensure the SQLite 3 gem is defined in your Gemfile
#   gem "sqlite3"

default: &default
  adapter: sqlite3
  pool: <%= ENV.fetch("RAILS_MAX_THREADS") { 5 } %>
  timeout: 5000

development:
  primary:
    <<: *default
    database: storage/development.sqlite3
  cache:
    <<: *default
    database: storage/development_cache.sqlite3
    migrations_paths: db/cache_migrate
  queue:
    <<: *default
    database: storage/development_queue.sqlite3
    migrations_paths: db/queue_migrate
  cable:
    <<: *default
    database: storage/development_cable.sqlite3
    migrations_paths: db/cable_migrate

test:
  primary:
    <<: *default
    database: storage/test.sqlite3
  cache:
    <<: *default
    database: storage/test_cache.sqlite3
    migrations_paths: db/cache_migrate
  queue:
    <<: *default
    database: storage/test_queue.sqlite3
    migrations_paths: db/queue_migrate
  cable:
    <<: *default
    database: storage/test_cable.sqlite3
    migrations_paths: db/cable_migrate

production:
  primary:
    <<: *default
    database: storage/production.sqlite3
  cache:
    <<: *default
    database: storage/production_cache.sqlite3
    migrations_paths: db/cache_migrate
  queue:
    <<: *default
    database: storage/production_queue.sqlite3
    migrations_paths: db/queue_migrate
  cable:
    <<: *default
    database: storage/production_cable.sqlite3
    migrations_paths: db/cable_migrate
```

Key points:
- All databases live in `storage/` directory (Rails 8 convention for SQLite)
- Development and test environments now get their own cache/queue/cable databases (previously only production had them)
- `timeout: 5000` prevents "database is locked" errors under concurrent access
- No `encoding: unicode` (PostgreSQL-specific)
- No `username`/`password`/`host` (SQLite is file-based)

**Important:** Since we are switching from PostgreSQL to SQLite, existing migrations cannot run against a fresh SQLite database because they use `t.jsonb` which is PostgreSQL-specific. The approach is:

1. **Drop the existing development and test databases** (they are PostgreSQL databases that can no longer be used)
2. **Update all existing migration files in-place** to change `t.jsonb` to `t.json` -- this ensures `db:migrate` works from scratch on a fresh SQLite database. The 5 migration files to update:
   - `db/migrate/20260327085826_create_agents.rb`: change `t.jsonb :adapter_config` to `t.json :adapter_config`
   - `db/migrate/20260327103011_create_audit_events.rb`: change `t.jsonb :metadata` to `t.json :metadata`
   - `db/migrate/20260327134948_create_heartbeat_events.rb`: change `t.jsonb :request_payload`, `t.jsonb :response_payload`, `t.jsonb :metadata` to `t.json` for all three
   - `db/migrate/20260327175602_create_notifications.rb`: change `t.jsonb :metadata` to `t.json :metadata`
   - `db/migrate/20260327191037_create_config_versions.rb`: change `t.jsonb :snapshot` and `t.jsonb :changeset` to `t.json` for both

3. **Run `bin/rails db:prepare`** -- this creates a fresh SQLite database and runs all migrations from scratch. Since SQLite is now the adapter, schema.rb will be regenerated with SQLite-native types.

4. **Verify the regenerated schema.rb**:
   - No `enable_extension "pg_catalog.plpgsql"` line
   - No `t.jsonb` columns -- all should be `t.json`
   - `bigint` columns become `integer` (SQLite uses integer for all integer types)
   - Partial index on invitations (`where: "(status = 0)"`) is preserved (SQLite supports partial indexes)
   - All foreign keys are preserved

5. **Run `bin/rails db:prepare RAILS_ENV=test`** to set up the test database

6. **Run `bin/rails test`** to verify all tests pass with SQLite

**Note on the partial index:** The invitations table has `where: "(status = 0)"` -- SQLite supports this syntax natively, so no change is needed.

**Note on fixtures:** All fixtures already use YAML hash syntax for jsonb columns (e.g., `metadata:` followed by indented key-value pairs), which works identically with json columns. No fixture changes needed.
