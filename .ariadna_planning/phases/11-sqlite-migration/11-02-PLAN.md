3. **Run `bin/bundler-audit`** -- verify no gem vulnerabilities (pg removal should not affect this)
4. **Run `bin/rails test`** -- run full test suite

**Known potential SQLite-specific issues to watch for:**

a. **jsonb vs json column type in tests**: If any test does raw SQL with `::jsonb` casting, it will fail. Grep for `::jsonb` in test files and fix. (Initial grep found none.)

b. **Concurrent access / locking**: SQLite has different locking behavior. The `timeout: 5000` in database.yml should handle most cases. If any test does parallel writes, it may need adjustment.

c. **Case sensitivity**: SQLite LIKE is case-insensitive by default (unlike PostgreSQL). If any test relies on case-sensitive LIKE, it may behave differently. (Unlikely to cause failures -- just different behavior.)

d. **Boolean handling**: SQLite stores booleans as 0/1 integers. Rails handles this transparently, but raw SQL in tests may need adjustment.

e. **HeartbeatScheduleManager**: Uses `defined?(SolidQueue::RecurringTask)` guard. Since SolidQueue now uses the same SQLite engine as primary (though separate database file), this should work. The guard was originally for checking if the solid_queue tables exist. Verify that the recurring task table is accessible.

f. **Partial index on invitations**: SQLite supports `WHERE` clauses in indexes. The `where: "status = 0"` syntax should work. Verify in generated schema.rb.

5. If ALL pass, run `bin/ci` for the full integrated pipeline.

**Fix strategy:**
- If a test fails due to PostgreSQL-specific SQL: rewrite to use ActiveRecord query interface
- If a test fails due to jsonb/json type difference: verify the column stores/returns Hash (not String)
- If a test fails due to locking: add retry or adjust test isolation
- Document any fixes in the summary
