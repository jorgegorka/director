# Milestone v1.5: Role Templates

**Status:** SHIPPED 2026-03-30
**Phases:** 26-28
**Total Plans:** 6

## Overview

Builtin department templates that let users bootstrap their AI company's org chart with pre-built hierarchies instead of creating every role manually. Five departments (Engineering, Marketing, Operations, Finance, HR) with 23 roles total, each with multi-paragraph job specs and skill pre-assignment.

## Phases

### Phase 26: Template Data and Registry
**Goal**: Ship 5 department YAML templates (Engineering, Marketing, Operations, Finance, HR) with a registry class that loads, validates, and exposes them
**Why this matters**: Users need a catalog of ready-made departments before they can apply anything -- the YAML files are the content that makes the feature valuable, and the registry is the foundation both the service and UI depend on.
**Depends on**: Phase 25 (v1.4 complete -- existing Role, Skill, and RoleSkill models in place)
**Requirements**: TMPL-01, TMPL-02, TMPL-03, SKILL-01
**Plans**: 2/2 complete

Plans:
- [x] 26-01: YAML template files and default_skills.yml extensions (template data)
- [x] 26-02: RoleTemplateRegistry class with validation and comprehensive tests

### Phase 27: Template Application Service
**Goal**: `ApplyRoleTemplateService` creates a complete department hierarchy with skill pre-assignment, skip-duplicate logic, and structured result reporting
**Why this matters**: This is the engine that turns a template definition into a real org chart -- without it, templates are just documentation. The service must be reliable enough that users trust one-click department creation.
**Depends on**: Phase 26 (templates must be loadable before they can be applied)
**Requirements**: APPLY-01, APPLY-02, APPLY-03, APPLY-04, APPLY-05
**Plans**: 2/2 complete

Plans:
- [x] 27-01: ApplyRoleTemplateService with hierarchy creation, skip-duplicate, skill assignment, result object
- [x] 27-02: ApplyAllRoleTemplatesService with CEO find-or-create and aggregate results

### Phase 28: Templates Browse and Apply UI
**Goal**: Dedicated templates pages where users can browse available departments, preview their role hierarchies, and apply them with one click
**Why this matters**: Users need to discover what templates are available and understand what they will get before committing -- the browse and preview pages build confidence, and the one-click apply makes the feature frictionless.
**Depends on**: Phase 27 (service must exist before the controller can delegate to it)
**Requirements**: UI-01, UI-02, UI-03, UI-04
**Plans**: 2/2 complete

Plans:
- [x] 28-01: RoleTemplatesController (index, show, apply), template card grid, hierarchy preview, flash feedback
- [x] 28-02: RoleTemplatesController tests and roles index discovery link

## Milestone Summary

**Key Decisions:**
- Data.define (Ruby 3.2+) for Template and TemplateRole value objects -- immutable, lightweight, named attributes
- No transaction wrapper for template application -- partial success preferred over all-or-nothing
- SQLite title column has no COLLATE NOCASE -- find_by(title:) is case-sensitive; skip-duplicate is title-exact-match only
- params[:id] is string template key, not integer DB ID -- RoleTemplateRegistry.find(key) does lookup
- Hierarchy tree rendered via recursive ERB lambda -- self-contained, avoids recursive partial complexity
- Reused ApplyRoleTemplateService::Result for ApplyAllRoleTemplatesService -- no new types needed
- No new gems, no migrations, no new models -- YAML files + plain Ruby registry + service object + standard controller

**Issues Resolved:**
- Parent ordering validation at load time prevents silent hierarchy corruption
- Tenant-scoped skill lookup (company.skills.where) prevents cross-tenant skill leakage
- button_to renders button[type=submit] not input[type=submit] -- caught in controller tests

**Technical Debt Incurred:**
- None
